缓存穿透、缓存击穿、缓存雪崩
Created2025-10-05|技术Redis
1. 缓存穿透：请求redis 里不存在的 key，此时会 穿过 redis 去访问 数据库。大量的请求导致数据库压力激增解决方案：redis 缓存设置空值，或者 布隆过滤器 2. 缓存击穿：一个 key是热点 key，redis 缓存里失效（过期）了（看起来就像被击穿了），导致大量请求被打到 数据库上。解决方案： 第一个去访问数据库的请求，为这个数据 设置互斥锁 setnx，后面的请求就不会去访问数据库了，而是返回一个默认值。 法 2：不设置真正的过期时间，而是自己实现逻辑过期时间，当一个请求发现过期时，他会去异步地数据库里更新数据。而由于物理上这个数据在 redis 里没有过期，其他请求会获取到旧数据。 3. 缓存雪崩：大量缓存击穿同时发生：比如系统初始化的时候，缓存被一起创建，如果他们的过期时间设置的一样，会导致大量数据在同一时间同时失效也可能是 redis 挂了，导致大量请求直接打到数据库解决方案： 随机设置过期时间对于 redis 挂了：设置哨兵集群

Redis怎么保证缓存与数据库一致性
Created2025-09-23|技术Redis
1. 为什么要直接删除缓存而非修改：删除是幂等的，修改不是==幂等==的。后面的值会覆盖前面的值，这样多次操作下结果顺序可能不一样。而删除的话，就能保证只有db里有数据，下次客户端缓存没命中的话，会去db里查询数据，然后再写回到redis 2. 既然已经确定了要删除缓存，那么问题就变成了 先删除缓存再修改db，还是先修改db再删除缓存？3. 删除缓存 + 修改db(不好)如果一个线程A要修改数据，先把redis删了。此时B发现缓存空了，就去db读了旧数据，然后把旧数据又写回redis了。此后，A才把正确的数据写入db，导致缓存db不一致。 4. 删除缓存 + 修改db + 延迟删除 （ 延迟双删）确保及时有别的线程在A的db没有更新成功的间隙读到了脏数据并写回了Redis，延迟双删可以保证这个 脏数据的 误写入 最终还是被 删除了 5. 修改db + 删除缓存（推荐）除非删除的操作执行失败了，需要设置一个MQ去监控发现失败了就重新去执行删除。其他情况是最好的
