缓存穿透、缓存击穿、缓存雪崩
1. 缓存穿透：请求redis 里不存在的 key，此时会 穿过 redis 去访问 数据库。大量的请求导致数据库压力激增解决方案：redis 缓存设置空值，或者 布隆过滤器 2. 缓存击穿：一个 key是热点 key，redis 缓存里失效（过期）了（看起来就像被击穿了），导致大量请求被打到 数据库上。解决方案： 第一个去访问数据库的请求，为这个数据 设置互斥锁 setnx，后面的请求就不会去访问数据库了，而是返回一个默认值。 法 2：不设置真正的过期时间，而是自己实现逻辑过期时间，当一个请求发现过期时，他会去异步地数据库里更新数据。而由于物理上这个数据在 redis 里没有过期，其他请求会获取到旧数据。 3. 缓存雪崩：大量缓存击穿同时发生：比如系统初始化的时候，缓存被一起创建，如果他们的过期时间设置的一样，会导致大量数据在同一时间同时失效也可能是 redis 挂了，导致大量请求直接打到数据库解决方案： 随机设置过期时间对于 redis 挂了：设置哨兵集群

websocket
工作流程websocket建立连接前会先使用 http协议发送 报文，在头部字段里说想要升级，服务端回复101后，双方的应用层改用websocket通信Tomcat的代码已经实现了处理http协议 和 ws协议的 切换 优势全双工，双方都可以向对方发起消息，利用了TCP的全双工能力，不像HTTP是半双工，同一时间只能有一个人发送数据，是请求-响应模式，服务端不能主动向客户端发送数据 数据包紧凑，开销比http小 适用于网络聊天，游戏，飞书这样的在线办公应用。服务端需要频繁地向客户端发送数据（不频繁的话可以用不停地发起HTTP请求的伪全双工）
