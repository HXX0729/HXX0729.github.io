JavaBean的编译时安全
JavaBean的编译时安全，本质上源于Java的静态类型检查机制。 Bean通过预定义的私有字段和公有的getter/setter方法，与编译器之间建立了一种 “强制契约”。相比于Map<String, Object>这类只有在运行时才能发现问题的弱类型容器，任何对Bean的非法操作，例如： 方法名拼写错误（如将setName错写为setNamme）。 传入参数类型不匹配（如为int类型的字段传入String）。 这些违反“契约”的行为，都会在编译阶段被编译器立即识别并报错，导致编译失败。 这种将错误检查从不可控的“运行时”提前到可控的“编译时”的机制，能从源头上杜绝大量因低级错误引发的NullPointerException和ClassCastException等运行时异常，从而显著提升代码的健壮性、稳定性和可维护性。 对比Map：1. 不安全的 Map 示例使用Map时，编译器无法检查Key的拼写或Value的类型，所有错误都只能在程序运行起来后才会暴露。 Java 123456789101112131415161718192021222324...

Ragflow 智能切片
传统切片：按照页数或者字数粗暴的切，比如 200 字一个 chunk 缺点：切片时可能没有考虑到语义信息，导致一个语义块被拆成两部分，那有的关键信息可能就无法被 RAG 召回 智能切片： 先对文档进行 cv，转化成 md 的形式 切分成自然语义块：根据 换行符，句号先分成一个个 chunk 。如果这个 chunk 的 token 比较少，再尝试和后面一个 chunk 合并，直到 token 数接近设置的阈值 如果遇到超大语义块：使用次级分隔符，比如：，；进行切割 ==亮点 ==：overlap 重叠机制：前后 2 个 chunk 会有一定重合，防止关键上下文丢失
